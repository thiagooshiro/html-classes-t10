Formulários e Responsividade

O que é um formulário HTML

Um formulário é um canal de comunicação entre o usuário e a aplicação. No projeto você tem dois: edit-profile-form (editar perfil) e new-card-form (criar novo local).

A tag <form> agrupa inputs relacionados e define onde e como os dados serão enviados.

Anatomia do formulário no projeto

edit-profile-form

  • input nome
  • input sobre mim
  • button[type=submit] Salvar

new-card-form

  • input título
  • input[type=url] link da imagem
  • button[type=submit] Criar

Atributos importantes dos inputs

type

Define o comportamento do campo. No projeto você tem type="text" e type="url". O type="url" já faz validação básica de URL nativamente — o browser rejeita o submit se o valor não começar com http:// ou https://.

text url email password number

name

É como o JS (e o backend) vai identificar o campo. No projeto: name="name", name="description", name="place-name", name="link". Sem name, o dado não é incluído no submit.

placeholder

Texto de dica que aparece quando o campo está vazio. Desaparece quando o usuário começa a digitar. Não substitui label — é só uma dica visual.

O projeto usa placeholder como substituto de label. Funcionalmente ok, mas acessibilidade fica comprometida — leitores de tela têm dificuldade. Em projetos reais, use <label>.

required

Impede o submit se o campo estiver vazio. No projeto, os campos do new-card-form têm required, os do edit-profile-form não têm. Percebeu a diferença de comportamento?

validação nativa sem JS necessário

O evento submit

Quando o usuário clica no button[type=submit], o formulário dispara o evento submit. O comportamento padrão do browser é recarregar a página — em SPAs você sempre previne isso:

// captura o formulário pelo id
const form = document.getElementById('edit-profile-form')

form.addEventListener('submit', (event) => {
  event.preventDefault() // impede o reload
  // lê os valores
  const name = form.querySelector('[name="name"]').value
})

Formulário dentro de popup

No projeto os formulários vivem dentro de .popup. Isso é um padrão comum — o formulário não está visível o tempo todo, é exibido via JS quando o usuário clica num botão.

O popup em si é um div com display: none no CSS. Quando o botão de editar é clicado, o JS adiciona uma classe que muda para display: flex ou display: block.

O formulário e o popup são responsabilidades separadas: o popup controla a visibilidade, o formulário controla os dados.

O que é responsividade

Responsividade é a capacidade do layout se adaptar a diferentes tamanhos de tela sem quebrar. O projeto roda em desktop, tablet e mobile — o mesmo HTML, comportamentos visuais diferentes.

A linha mais importante do projeto pra responsividade nem está no CSS — está no HTML:

<meta name="viewport" content="width=device-width, initial-scale=1.0">

Sem isso, mobile renderiza a página no tamanho de desktop e diminui tudo. Com isso, o browser usa a largura real do dispositivo.

Media queries

Media queries são condições CSS. "Se a tela for menor que X, aplica essas regras."

/* layout padrão — desktop */
.cards__list {
  grid-template-columns: repeat(3, 1fr);
}

/* tela menor que 768px — tablet/mobile */
@media (max-width: 768px) {
  .cards__list {
    grid-template-columns: 1fr;
  }
}

Breakpoints comuns

  • 320px — mobile pequeno
  • 480px — mobile grande
  • 768px — tablet
  • 1024px — desktop pequeno
  • 1280px — desktop
  • 1440px — desktop grande

No projeto Around the U.S., o breakpoint principal costuma ser em torno de 768px — onde o layout de cards e o perfil mudam de horizontal pra vertical.

Mobile-first vs desktop-first

Desktop-first

Escreve o layout desktop no CSS base, usa max-width pra adaptar pra telas menores.

mais comum em projetos legados

Mobile-first

Escreve o layout mobile no CSS base, usa min-width pra expandir pra telas maiores.

recomendado hoje

O projeto Around the U.S. foi projetado desktop-first — você vai encontrar max-width nas media queries.

Elementos que mais mudam em mobile

No projeto especificamente:

.profile .cards__list .page__content .popup__content .profile__title
  • .profile — vira coluna em vez de linha
  • .cards__list — de grid multi-coluna pra coluna única
  • .page__content — padding lateral reduz
  • .popup__content — largura máxima vira 100%
  • .profile__title — font-size diminui

Unidades responsivas

Relativas — se adaptam

  • % — relativo ao pai
  • vw / vh — relativo à viewport
  • rem — relativo ao root (16px)
  • em — relativo ao elemento pai

Absolutas — não se adaptam

  • px — pixel fixo
  • Use pra bordas, gaps internos, font-size fixo.
  • Evite pra larguras de container.
